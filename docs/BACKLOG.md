## 未解決課題一覧

### 1. 🔴 Netlify設定がリポジトリ外にある（引き継ぎリスク）
- **現状**: 本番デプロイ設定（公開ディレクトリ・対象ブランチ`production`・環境変数）が `netlify.toml` に無く、**Netlify管理画面にのみ存在**する。
- **影響**: 管理画面にアクセスできない担当者には本番構成が見えない。複数管理体制での最大の引き継ぎリスク。
- **対応の方向性**: 
  - (a) 管理画面の設定内容を文書化して共有（短期・すぐ可能）
  - (b) `netlify.toml` をリポジトリに追加し、設定をコード管理化（恒久対策）
- **メモ**: `.github/workflows` も無く、デプロイはビルド無しの静的配信。詳細は `CLAUDE.md` の Deployment Safety Model 参照。
- **最終更新: 2026-07-11**

### 2. 🔵 プッシュ通知が端末に表示されない
- **現状**: FCMの送信自体は成功するが、端末側で通知が表示されない。回避策として**メール通知方式を実装済み**（現行のリマインダーはメール）。
- **影響**: プッシュ通知機能は事実上未稼働。ただしメールで代替できているため実運用への支障は小さい。
- **対応の方向性**: 
  - (a) 根本原因を再調査して復活させる
  - (b) プッシュ通知を**正式に廃止**し、関連コード（`firebase-messaging-sw.js`, FCMトークン処理等）を整理する
  - → **どちらに倒すか方針決定が必要**
- **参照**: `docs/PUSH_NOTIFICATION_INVESTIGATION.md`
- **最終更新: 2026-07-11**

### 3. 🟢 App Check が無効化されている（※方針転換により緊急性消滅）
- **現状**: パフォーマンステストのため一時的に無効化中（`js/firebase-config.js` 内でコメントアウト）。
- **影響**: Firestore/Auth への不正アクセスに対する防御レイヤーが1枚無い状態。
- **⚠→🟢 方針転換（2026-07-12）**: **Firebaseを主体とせず、ローカル完結（IndexedDB）主体へ全面移行**する方針が確定（`docs/specs/安心利用_協力利用_ログイン設計仕様書.md` 11章）。これに伴い、公開ローンチ前の App Check 再有効化という緊急課題は**不要化**した。協力利用（Firestore）が救済措置として残る過渡期のみ低優先で意識すればよい。移行完了後はクローズ予定。
- **最終更新: 2026-07-12**

### 4. 🟡 写真がFirestoreにbase64で保存されている
- **現状**: 人物写真を Cloud Storage ではなく Firestore ドキュメント内に base64 文字列で保存している。
- **この課題は2軸ある（混同しないこと）**:
  - **① スケーラビリティ（容量）**: 写真が増えるとドキュメントサイズ・読み込み量が増大し、スケールしにくい。Firestoreの1ドキュメント上限（1MB）にも注意。→ 本項目の主眼。
  - **② プライバシー（保護対象外）**: base64は暗号化ではなく、管理者はコンソールで写真を閲覧可能。新設計（安心利用/協力利用）では**写真は保護対象に含めない**ことを確定済み（協力利用の同意文言で正直に開示）。→ セキュリティ課題として `#3 App Check` と併せて認識する。
- **対応の方向性**: (①) Cloud Storage への移行を検討。移行時は既存データのマイグレーションが必要（`MIGRATION.md` の手法が参考になる）。(②) 保護が必要になった場合は、暗号化はネイティブアプリ化時に検討（設計仕様書 7章参照）。
- **参照**: `docs/specs/安心利用_協力利用_ログイン設計仕様書.md`
- **最終更新: 2026-07-11**

### 5. 🟢 オフライン永続化が複数タブで失敗する
- **現状**: Firestoreのオフライン永続化（`enablePersistence`）が、複数タブを同時に開くと `failed-precondition` エラーになる。
- **影響**: 複数タブ利用時に永続化が効かない。単一タブ利用なら問題なし。実害は限定的。
- **対応の方向性**: マルチタブ対応の永続化設定（`synchronizeTabs`）の採用を検討。ただし副作用の検証が必要。
- **最終更新: 2026-07-11**

### 6. 🟡 migrateツールのキー指定を堅牢化する
- **現状**: `tools/migrate-sortorder.js` はサービスアカウントキーを**特定のファイル名でハードコード**（`...fbsvc-8166a79bf5.json`）して読み込んでいる。
- **問題**: Firebaseの管理者キーはダウンロード毎に**ランダムな符号が付く**ため、キーを再発行するとこの固定名と一致せず「ファイルなし」で止まる。**複数管理体制では各担当が別々の名前のキーを発行するため、ほぼ確実に踏む**（1人運用でも再発行すれば発生）。
- **対応の方向性**: キー名のハードコードをやめ、(a) 環境変数でパスを渡す、または (b) `serviceAccountKey.json` など固定の決まった名前に統一する。
- **メモ**: これは「場所」ではなく「名前」の問題。ファイルの`tools/`移動に伴う場所（相対パス）は `../` 修正で対応済み。Firebaseの仕様と現行コードの噛み合わせに起因。
- **最終更新: 2026-07-11**

### 8. 🔴 安心利用（ゲスト/IndexedDB）がFirestoreへ書き込まないことの担保
- **現状**: 新設計では「安心利用」を主軸に据え、「運営者を含め誰も内容を見られない」と案内する予定。この保証はゲストデータが**Firestoreへ一切書き込まれない**ことに100%依存する。
- **影響**: もし安心利用のいずれかの経路でFirestoreへデータが漏れると、**設計の看板（"安心"）が崩れる**。最重要の正しさ保証。
- **対応の方向性**: 実装セッションで `db-local.js` / `auth.js` の分岐を検証し、ゲスト状態からFirestoreへ到達する経路が無いことをコードで確認。あわせて Firebase Console で「安心利用操作時に書き込みが発生しない」ことを実測。
- **参照**: `docs/specs/安心利用_協力利用_ログイン設計仕様書.md`（8章チェックリスト）
- **最終更新: 2026-07-11**
