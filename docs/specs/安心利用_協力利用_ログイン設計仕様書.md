# 「安心利用」「協力利用」ログイン設計仕様書

**作成日：** 2026年7月11日
**作成者：** 菅原克行（壁打ちチャットにて設計・Claude Code実装セッションへ引き継ぎ）
**対象：** 美点発見note（本番：https://biten-note.netlify.app）
**目的：** 実践会経由での新規利用者（2週間後の講習会で100名規模を想定）に向けて、登録不要で安心して使える導線を整備する
**参照：** biten_note_システム構成図_20260711.pdf（Firebaseプロジェクト構成・デプロイフロー）

---

## 1. 背景・経緯

当初、Firestoreに保存されるデータ（名前・美点内容）を管理者も含めて誰も読めない状態にするため、Web Crypto APIを使った「合言葉方式」の暗号化実装を検討していた。

しかし壁打ちの中で以下が判明し、方針を転換した。

- 合言葉から生成した鍵はlocalStorage/sessionStorageに保存できない（CLAUDE.mdのルールに準拠）ため、**タブを閉じるたびに合言葉の再入力が必要**になり、UXの手間が大きい
- 実装・既存ユーザーへの回帰テストのリスクが、2週間という期限に対して重い
- 一方で、**ゲストモード（IndexedDB・端末内完結）はすでに実装済み**であり、これを使えば「本人以外は誰も読めない」状態がノーコストで実現できる

これを受けて、**今回のスコープでは合言葉方式の暗号化実装は行わない**。代わりに、既存のゲストモードを前面に押し出し、Firestore同期（登録）を「開発協力」として正直に位置づけ直す方向に設計を切り替えた。

---

## 2. 名称の決定

| 旧名称 | 新名称 | 内容 |
|---|---|---|
| ゲストモード | **安心利用** | 登録不要。IndexedDBでこの端末・このブラウザにのみ保存。運営者を含め誰も内容を見ることができない |
| （登録・Firestore同期） | **協力利用** | メールまたはGoogleで登録。Firestoreに同期され、運営者が開発改善のために内容を確認できる状態で保存される |

選定理由：実践会の会員が自分の大切な人に紹介する際、文字を読んだだけで意味が伝わる分かりやすさを最優先した（「安心利用」「協力利用」）。

## 3. 画面フロー設計

壁打ちの中でさらに検討が進み、**最終的に「安心利用ボタンのみを前面に出し、協力利用への入口はフッターの小さなリンクに留める」形に確定**した。

### 3.1 決定に至った経緯

当初は「安心利用を主・協力利用をリンクで併記」という案A構成だったが、以下の理由でさらに一歩踏み込んだ。

- Firestoreの設定（データが運営者から見える構造であること）を正直に伝えようとすると、**Netlify本番であえて新規登録の入口を積極的に見せる必然性は薄い**と判断
- 複数端末での利用ニーズが今後増えてきた場合は、暗号化と同様に**ネイティブアプリ化のタイミングで改めて検討**する方針とし、現時点は安心利用を標準とする
- ただし、**既存21名以上の登録ユーザーのログイン導線は、本番（Netlify）に残す必要がある**（この点は壁打ち中に見落としかけていたため、要注意ポイントとして記録）

### 3.2 確定した画面構成

```
ログイン画面
　↓
[ 安心利用（登録不要）ではじめる ]  ← 唯一の主ボタン（大きく・上部）
　入力データはこの端末だけに保存されます。  ← 補足テキスト
　
　　　　　　（余白）
　
－－－－－－－－－－－－－－－－－－  ← フッター
以前からご利用の方はこちら（開発・テスト用）  ← 小さいリンク
　→ タップすると既存のログイン／新規登録フォームへ
　→ 新規登録の場合、同意文言＋チェックボックスを表示（4章参照）
```

**確定した画面文言：**

| 要素 | 文言 |
|---|---|
| 主ボタン | 安心利用（登録不要）ではじめる |
| ボタン下の補足テキスト | 入力データはこの端末だけに保存されます。 |
| フッターリンク | 以前からご利用の方はこちら（開発・テスト用） |

### 3.3 フッターリンクの位置づけ

- **既存ユーザー**：このリンクから今まで通りログイン可能。表示・機能とも変更なし
- **新規訪問者（実践会経由）**：フッターの小さな文言（「開発・テスト用」）により、興味本位でのクリックを意図的に抑制する。既存ユーザーはこれまで口頭でテスト利用への理解を得て登録しているため、この文言表現で問題ないと判断済み
- 万一、既存ユーザーがログアウト後に誤って安心利用ボタンを押した場合：**Firestore上の元データが削除されることはない**（安心利用はIndexedDBという別領域に新規のまっさらな状態で入るだけ）。データが見当たらず困った場合は問い合わせ対応で個別に案内する運用とする

### 3.4 実装への影響

- 安心利用側にはログイン処理そのものが発生しないため、既存のIndexedDB実装への変更は基本的に不要
- フッターリンクの遷移先は、既存の登録・ログインフォームをそのまま使う（新規作成ではなく再配置）
- 協力利用の新規登録フローに、同意ステップを1つ追加する点は変更なし（4章参照。ただし到達経路がフッター経由になったため、実際に到達する新規訪問者は限定的になる見込み）

## 4. 協力利用の同意文言（画面表示イメージ）

トーンは壁打ちで**「事実ベース・淡々とした説明」を維持する**方向に確定（感謝の言葉を添える案・短文化する案は不採用）。箇条書き構成もこのまま維持する。

```
━━━━━━━━━━━━━━━━━━━
協力利用について

美点発見noteの改善にご協力いただける方向けの
登録方法です。

・お名前や美点の内容は、開発改善のため
  運営者が確認できる状態で保存されます
・お写真は現在、保護対象に含まれておりません
  （運営者が閲覧できる状態で保存されます）
・データは複数の端末で同期してご利用いただけます

□ 上記に同意して登録する

[登録する]
━━━━━━━━━━━━━━━━━━━
```

## 5. 安心利用（IndexedDB）についての正確な説明

案内資料・画面文言で使う際は、以下の**正確な挙動**に基づいて記述すること（不正確な表現は使わない）。

**誤り（使わない）：**
- 「電源を切ると消えます」

**正しい説明：**
- この端末・このブラウザに保存され、**タブを閉じても、電源を切っても消えない**
- 消えるのは以下の場合のみ：
  - 本人がブラウザの「サイトデータを削除」を行った場合
  - ブラウザ自体をアンインストールした場合
  - 別の端末・別のブラウザで開いた場合（データは引き継がれない）
  - 一部のブラウザ（特にiOS Safari等）で長期間サイトを開かない場合、自動削除される可能性がある

## 5.5 Firebase利用枠についての注意（システム構成図より）

システム構成図（2026年7月11日版）の確認により、Firebaseプロジェクト（biten-note-app）は**Sparkプラン（無料枠のみ）ではなくBlazeプラン（従量課金プラン）**であることが判明。Blazeプランは無料枠を超えると自動的に課金が発生する点がSparkプランと異なる。

100名規模が短期間に流入する今回のケースでは、特にCloud Functions（毎時0分起動のメールリマインダー関数）とFirestoreの読み書きが対象になる。実害の可能性は高くないと考えられるが、**講習会前にFirebase Console（プロジェクト設定→使用量）で現在の使用量を確認しておく**ことをタスクとして追加する。正確な数値は本文書内では確認できないため、実際にConsoleで確認すること。

## 6. 既存システムへの影響範囲

| 箇所 | 影響 |
|---|---|
| db.js | 変更なし想定（安心利用は既存IndexedDB実装をそのまま使用、協力利用は既存Firestore実装をそのまま使用） |
| ログイン画面（index.html / app.js） | 表示の変更。主ボタンを「安心利用」のみとし、既存の登録・ログイン導線はフッターの小さなリンクへ移動 |
| 登録フロー（auth.js） | 協力利用の新規登録時、登録完了前に同意チェックボックスのステップを追加（フッター経由でのみ到達） |
| 既存ユーザー | **ログイン機能自体への影響なし**。フッターのリンクから今まで通りログイン可能。画面上の見え方（主ボタンの位置）が変わるのみ |
| プライバシーポリシー | 「安心利用」「協力利用」それぞれの実態に合わせて文言を更新する必要あり（別途対応） |

## 8. Claude Codeセッションでの作業チェックリスト（想定）

```
□ ログイン画面のUI変更（安心利用ボタンのみを主表示、既存の登録・ログイン導線をフッターへ移動）
□ フッターリンクの実装・遷移先確認（既存の登録・ログインフォームへの接続、表示崩れがないか）
□ 協力利用（フッター経由の新規登録）選択時の同意チェックボックス画面の追加
□ 同意文言の最終コピー確定（4章のドラフトをベースに調整）
□ 動作確認：安心利用でFirestoreへの書き込みが一切発生しないことをFirebase Consoleで確認
□ 動作確認：協力利用の登録フローが同意ステップを含めて問題なく完了するか確認
□ 動作確認：既存ユーザーのログイン・利用に影響が出ていないか確認（回帰テスト）
□ GitHub Pages（開発環境）でテスト → 問題なければ Netlify（本番）へデプロイ
□ Firebase Console（プロジェクト設定→使用量）で、講習会前にBlazeプランの使用量を確認
□ テスト用アカウント（例：test+encryption1@...等、判別しやすいメールアドレス）でのみ動作確認し、テスト後はFirebase Consoleから手動で削除
```

※開発環境（GitHub Pages）と本番環境（Netlify）は**同じFirebaseプロジェクトを共有**している（システム構成図より確認済み）。開発環境でのテストも本番と同じFirestoreに書き込まれるため、テストアカウントの後始末を徹底すること。`tools/migrate-*`等の管理スクリプトも本番Firestoreを直接書き換える点に注意。
