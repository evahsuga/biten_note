## 未解決課題一覧

### 1. 🔴 Netlify設定がリポジトリ外にある（引き継ぎリスク）
- **現状**: 本番デプロイ設定（公開ディレクトリ・対象ブランチ`production`・環境変数）が `netlify.toml` に無く、**Netlify管理画面にのみ存在**する。
- **影響**: 管理画面にアクセスできない担当者には本番構成が見えない。複数管理体制での最大の引き継ぎリスク。
- **対応の方向性**: 
  - (a) 管理画面の設定内容を文書化して共有（短期・すぐ可能）
  - (b) `netlify.toml` をリポジトリに追加し、設定をコード管理化（恒久対策）
- **メモ**: `.github/workflows` も無く、デプロイはビルド無しの静的配信。詳細は `CLAUDE.md` の Deployment Safety Model 参照。
- **最終更新: 2026-07-11**

### 2. 🔵 プッシュ通知が端末に表示されない
- **現状**: FCMの送信自体は成功するが、端末側で通知が表示されない。回避策として**メール通知方式を実装済み**（現行のリマインダーはメール）。
- **影響**: プッシュ通知機能は事実上未稼働。ただしメールで代替できているため実運用への支障は小さい。
- **対応の方向性**: 
  - (a) 根本原因を再調査して復活させる
  - (b) プッシュ通知を**正式に廃止**し、関連コード（`firebase-messaging-sw.js`, FCMトークン処理等）を整理する
  - → **どちらに倒すか方針決定が必要**
- **参照**: `docs/PUSH_NOTIFICATION_INVESTIGATION.md`
- **最終更新: 2026-07-11**

### 3. 🟢 App Check が無効化されている（※方針転換により緊急性消滅）
- **現状**: パフォーマンステストのため一時的に無効化中（`js/firebase-config.js` 内でコメントアウト）。
- **影響**: Firestore/Auth への不正アクセスに対する防御レイヤーが1枚無い状態。
- **⚠→🟢 方針転換（2026-07-12）**: **Firebaseを主体とせず、ローカル完結（IndexedDB）主体へ全面移行**する方針が確定（`docs/specs/安心利用_協力利用_ログイン設計仕様書.md` 11章）。これに伴い、公開ローンチ前の App Check 再有効化という緊急課題は**不要化**した。協力利用（Firestore）が救済措置として残る過渡期のみ低優先で意識すればよい。移行完了後はクローズ予定。
- **最終更新: 2026-07-12**

### 4. 🟡 写真がFirestoreにbase64で保存されている
- **現状**: 人物写真を Cloud Storage ではなく Firestore ドキュメント内に base64 文字列で保存している。
- **この課題は2軸ある（混同しないこと）**:
  - **① スケーラビリティ（容量）**: 写真が増えるとドキュメントサイズ・読み込み量が増大し、スケールしにくい。Firestoreの1ドキュメント上限（1MB）にも注意。→ 本項目の主眼。
  - **② プライバシー（保護対象外）**: base64は暗号化ではなく、管理者はコンソールで写真を閲覧可能。新設計（安心利用/協力利用）では**写真は保護対象に含めない**ことを確定済み（協力利用の同意文言で正直に開示）。→ セキュリティ課題として `#3 App Check` と併せて認識する。
- **対応の方向性**: (①) Cloud Storage への移行を検討。移行時は既存データのマイグレーションが必要（`MIGRATION.md` の手法が参考になる）。(②) 保護が必要になった場合は、暗号化はネイティブアプリ化時に検討（設計仕様書 7章参照）。
- **参照**: `docs/specs/安心利用_協力利用_ログイン設計仕様書.md`
- **最終更新: 2026-07-11**

### 5. 🟢 オフライン永続化が複数タブで失敗する
- **現状**: Firestoreのオフライン永続化（`enablePersistence`）が、複数タブを同時に開くと `failed-precondition` エラーになる。
- **影響**: 複数タブ利用時に永続化が効かない。単一タブ利用なら問題なし。実害は限定的。
- **対応の方向性**: マルチタブ対応の永続化設定（`synchronizeTabs`）の採用を検討。ただし副作用の検証が必要。
- **最終更新: 2026-07-11**

### 6. 🟡 migrateツールのキー指定を堅牢化する
- **現状**: `tools/migrate-sortorder.js` はサービスアカウントキーを**特定のファイル名でハードコード**（`...fbsvc-8166a79bf5.json`）して読み込んでいる。
- **問題**: Firebaseの管理者キーはダウンロード毎に**ランダムな符号が付く**ため、キーを再発行するとこの固定名と一致せず「ファイルなし」で止まる。**複数管理体制では各担当が別々の名前のキーを発行するため、ほぼ確実に踏む**（1人運用でも再発行すれば発生）。
- **対応の方向性**: キー名のハードコードをやめ、(a) 環境変数でパスを渡す、または (b) `serviceAccountKey.json` など固定の決まった名前に統一する。
- **メモ**: これは「場所」ではなく「名前」の問題。ファイルの`tools/`移動に伴う場所（相対パス）は `../` 修正で対応済み。Firebaseの仕様と現行コードの噛み合わせに起因。
- **最終更新: 2026-07-11**

### 8. ✅ 完了：安心利用（ゲスト/IndexedDB）がFirestoreへ書き込まないことの担保
- **保証内容**: 「安心利用」を主軸に据え「運営者を含め誰も内容を見られない」と案内する前提。ゲストデータが**Firestoreへ一切書き込まれない**ことに100%依存する、最重要の正しさ保証。
- **検証結果（2026-07-12 完了）**: 静的コード監査（全数調査＋実読）により、**安心利用（ゲスト・未ログイン）から Firestore へ書き込まれる経路は存在しない**ことを確認。3層防御＝①`App.getDB()`(app.js:46) がゲスト中は必ず LocalDB を返す ②`DB.getCurrentUserId()`(db.js:140) が未ログイン時 throw ③`firestore.rules` が `request.auth != null` 要求（匿名認証は未使用）。唯一のゲスト由来書込は「ログイン確定＋confirm同意後の移行」のみ。
- **第4層ガードの扱い**: 実行時強制ガードは**あえて入れない**（現行コードで発火せず移行フローへ不要な結合が生じるため）。代わりに `biten_note/CLAUDE.md` に不変条件・変更時注意・引き継ぎ時のガード導入検討を明記。
- **Console確認**: 3層防御がコードで確認できたため、実機/Firebase Console 実測は方針により省略。
- **記録**: `docs/notes/検証記録_安心利用Firestore非書込_20260712.md`
- **最終更新: 2026-07-12**

### 11. 🟡 registerFcmToken が公開・認証未検証（サーバ側）
- **発見（2026-07-12、#8監査の副産物）**: Cloud Function `registerFcmToken`（`functions/index.js:557`）が `invoker:"public"` かつ認証トークン未検証で、リクエストボディの `userId` をそのまま使う。任意の `userId` の `users/{userId}/settings/notifications` に fcmToken を書ける可能性がある。
- **影響**: 他人の通知設定ドキュメントへの書き込み・上書きの余地。ただしプッシュ通知自体が現状未稼働（#2）で、メール通知に代替済みのため実害は限定的。
- **対応の方向性**: (a) Firebase ID トークン検証を追加し `userId` を検証済みトークンから取得する、または (b) プッシュ通知廃止方針（#2）と合わせてエンドポイント自体を撤去。**#8（安心利用のFirestore非書込）とは別問題**。
- **最終更新: 2026-07-12**

### 12. 🟢 handleRedirectResult がゲストフラグを解除しない（潜在）
- **発見（2026-07-12、#8監査の副産物）**: `Auth.handleRedirectResult()`（`js/auth.js:68`）はログイン成功時に `createUserDocument` は呼ぶが `exitGuestMode()`／`handleGuestDataOnLogin()` を呼ばない。もし将来 Google ログインを `signInWithRedirect` 方式に変更すると、ログイン済みなのに `bitenNote_guestMode` が残る stale flag 状態になり得る。
- **現状**: `signInWithRedirect` は未使用（Googleはポップアップ方式のみ）のため、この成功ブランチには到達せず**無害**。
- **対応の方向性**: リダイレクト方式を有効化する際は、成功ブランチで `handleGuestDataOnLogin(uid)`（＝フラグ解除＋移行）をポップアップ経路と同様に呼ぶこと。`biten_note/CLAUDE.md` の不変条件セクションにも注意を記載済み。
- **最終更新: 2026-07-12**

### 13. 🟡 協力利用の同意まわりの残作業（Google経路の同意／§4文言の最終確定）※設計者と壁打ち
- **発見（2026-07-12、同意チェック精査より）**: メール/パスワード新規登録の同意ゲートは**実装済み**（`#agreeTerms` required チェックボックス＋`handleSignup` app.js:3182-3189 のJS二重確認＋「開発協力について」説明ボックス app.js:2934）。以下2点が残課題。
- **① Google経路に当サービスの同意ステップが無い**: Googleボタンは signupForm 外の共用（app.js:3024付近）で `agreeTerms` を通らず、`handleGoogleLogin`（app.js:3210）も同意確認なし。表示される「初回ログインの方へ」やGoogleの英語画面は**Google連携の同意**であり、当サービスのPP/利用規約への同意ではない。→ 新規Googleユーザーは規約に明示同意せず登録できる。
- **② 同意文言が仕様§4ドラフトと不一致**: 現状は「写真は…分析に利用できる状態で保存」だが、§4は「お写真は**保護対象に含まれておりません**（運営者が閲覧できる状態）」とより直接的。チェックボックス文言も差異。仕様§9で「文言は未確定・最終コピーは別途確定」とされる設計者マター。
- **対応の方向性**: ①のGoogle同意付与（ワンクリック導線へのチェック噛ませ等のUX設計が必要／ログイン・登録の出し分けも要検討）と、②の文言確定を**セットで設計者と壁打ち**して決める。協力利用は将来サンセット予定・新規は導線を絞っているため優先度は中。
- **最終更新: 2026-07-12**
