# 作業ログ（WORKLOG）

開発の**進捗・主要な決定・学び・次への申し送り**を時系列で残す台帳。複数管理体制での引き継ぎと、次の展開に活かすことが目的。

**書き方**
- 作業のまとまり（スプリント／セッション）ごとに、**先頭に**エントリを追加（新しい順）。
- 各エントリのテンプレート：**やったこと** ／ **主要な決定・学び（なぜ）** ／ **次への申し送り**。
- 役割分担：課題は `BACKLOG.md`、デプロイの実態は `DEPLOYMENT_NOTES.md`、設計仕様は `docs/specs/`。ここは「経緯と学びの物語」を担う。

### 主要な決定・学び（なぜ）
- **安心利用の非書込**は「ルーティング(getDB)＋DB層の未ログインthrow＋firestore.rules」の3層で担保済み。実行時ガードは現行コードで発火せず移行フローに不要な結合を生むため**オーバースペックと判断**→コード化せず CLAUDE.md 明文化＋引き継ぎ時の導入検討に留めた。
- **Google 同意**はワンクリックで認証が完了する仕組み上、事前チェックでは止められない → **認証直後の必須モーダル**で担保。新規判定は SDK の `additionalUserInfo.isNewUser` に依存せず **Firestore ドキュメント有無**が確実（穴を塞ぐ）。
- **クロスオリジンは IndexedDB を共有できない**（GitHub Pages ↔ Netlify、別端末）。データ移行のエクスポート/インポートは今回**見送り**（BACKLOG #14）。既存 GitHub ゲストは継続利用でOK、移行準備完了後に**お知らせ機能で案内**する作戦。
- **GitHub Pages は最大10分キャッシュ**（`Cache-Control: max-age=600`）。更新直後に実機で反映されない時はシークレットタブ/待機で確認。
- **昇格の切り分け**：Cloud Functions（メール）は**未デプロイ**（`firebase deploy --only functions` を実行しない）。通知UIはホスト判定で本番自動非表示。
- **お知らせの新着判定は「id 差分（per-user）」**。id は安定・一意・不変が鉄則（変更=全員再通知／再利用=非通知）。
